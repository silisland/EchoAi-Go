## 硬件对比概览

### RK3562 vs RV1106 规格对比

| 规格项 | RK3562 | RV1106 | 差异说明 |
|--------|--------|--------|----------|
| **CPU** | 4× Cortex-A53 @ 2.0GHz | 1× Cortex-A7 @ 1.5GHz | **4核 vs 单核**，8.5倍性能提升 |
| **架构** | ARMv8-A (64位) | ARMv7-A (32位) | 64位系统支持 |
| **NPU** | 1 TOPS | 0.5 TOPS | **2倍性能**，支持更多数据类型 |
| **NPU 精度** | INT4/INT8/INT16/FP16/BF16 | INT8/INT4/INT16 | FP16/BF16 支持更好精度 |
| **GPU** | Mali-G52 2EE (3D) | 2D 图形引擎 | 支持3D加速，LVGL动画更流畅 |
| **内存** | 最大8GB DDR4/LPDDR4X | 嵌入式 512Mb-2Gb | **4-16倍内存容量** |
| **内存位宽** | 32位 | 16位 | 双倍内存带宽 |
| **视频编解码** | 4K@30fps 解码 | 4K@25fps 编码 | 更高的视频处理能力 |
| **接口** | PCIe 2.1, GbE, CAN | USB, 百兆网 | 更丰富的扩展接口 |
| **功耗** | ~5W | ~2W | 功耗增加，需更好散热 |
| **封装** | FCBGA478L (13.9×13.9mm) | QFN88 (9×9mm) | 更大封装，PCB重新设计 |

### 关键升级点

**✅ RK3562 优势：**
- **4核CPU** = AI 服务端可以并行处理 VAD + ASR + Chat + TTS
- **1 TOPS NPU** = YOLOv5 推理速度翻倍，支持 FP16 精度更高
- **8GB内存** = 可以加载更大的 AI 模型，不再受内存限制
- **GbE网口** = 网络传输更快，AI 响应延迟更低
- **PCIe 2.1** = 可扩展 NVMe SSD 或高速外设

**⚠️ 需要注意：**
- **功耗增加** = 从 2W 到 5W，需要更好的电源和散热
- **封装变大** = PCB 需要重新设计，不能直接用原板
- **64位系统** = 需要重新编译工具链和库
- **价格更高** = BOM 成本增加

## 迁移优势分析

### 为什么 RK3562 更适合 Echo-Mate

#### 1. AI Pipeline 性能提升

**原 RV1106 瓶颈：**
```
单核 Cortex-A7 + 0.5 TOPS NPU
├── VAD 检测: CPU 占用 20%
├── ASR 识别: CPU 占用 40% + NPU 占用 30%
├── LLM 对话: CPU 占用 60% (调用云端 API，本地处理)
└── TTS 合成: CPU 占用 30%
总计: 单核 150% 负载 → 严重卡顿，无法并行
```

**RK3562 改进：**
```
4核 Cortex-A53 + 1 TOPS NPU
├── VAD 检测: Core 1, CPU 15%
├── ASR 识别: Core 2, CPU 25% + NPU 20%
├── LLM 对话: Core 3, CPU 30%
└── TTS 合成: Core 4, CPU 20%
总计: 4核并行，每核 30% 负载 → 流畅运行
```

#### 2. YOLOv5 目标检测优化

| 指标 | RV1106 | RK3562 | 改进 |
|------|--------|--------|------|
| **推理延迟** | ~200ms | ~80ms | 2.5倍更快 |
| **模型精度** | INT8 (量化损失) | FP16 (接近原始) | 准确率 +5-10% |
| **支持模型大小** | < 50MB | < 200MB | 可加载更大模型 |
| **并发检测** | 1路 | 2-3路 | 支持多摄像头 |

#### 3. LVGL GUI 体验提升

**Mali-G52 GPU 带来的改进：**
- **硬件加速渲染**: 复杂的渐变、阴影效果不卡顿
- **3D 动画支持**: 可以实现卡片翻转、立方体切换等效果
- **视频播放**: 可以直接在 UI 中播放视频
- **GPU 内存**: GPU 和 CPU 共享内存，不再互相抢占

## 技术栈差异

### 1. 编译工具链变化

**RV1106 (32位 ARM):**
```bash
# 交叉编译器
sudo apt install gcc-arm-linux-gnueabihf

# CMake 配置
cmake .. -DTARGET_ARM=ON \
  -DCMAKE_C_COMPILER=arm-linux-gnueabihf-gcc \
  -DCMAKE_CXX_COMPILER=arm-linux-gnueabihf-g++
```

**RK3562 (64位 ARM):**
```bash
# 新的交叉编译器
sudo apt install gcc-aarch64-linux-gnu

# CMake 配置
cmake .. -DTARGET_ARM=ON \
  -DCMAKE_SYSTEM_PROCESSOR=aarch64 \
  -DCMAKE_C_COMPILER=aarch64-linux-gnu-gcc \
  -DCMAKE_CXX_COMPILER=aarch64-linux-gnu-g++ \
  -DCMAKE_SYSTEM_NAME=Linux
```

### 2. RKNN 版本差异

| 特性 | RV1106 | RK3562 |
|------|--------|--------|
| **RKNN Toolkit** | V1.7.5+ | V1.7.5+ (兼容) |
| **NPU 驱动** | rknn_driver V2 | rknn_driver V2 |
| **API 兼容性** | 相同 | 相同 |
| **量化方式** | INT8 | INT8/FP16/BF16 |
| **模型转换** | 相同流程 | 相同流程 |

**好消息**: RKNN API 完全兼容，RV1106 的代码可以直接用！

### 3. 内核和驱动差异

**Linux 内核:**
- RV1106: Linux 4.19 / 5.10
- RK3562: Linux 5.10 (Rockchip SDK)

**关键驱动差异:**
```c
// 设备树配置变化
// RV1106
compatible = "rockchip,rv1106";

// RK3562
compatible = "rockchip,rk3562";

// DDR 配置
// RV1106: 内置 SIP DDR，无需配置
// RK3562: 外置 DDR4，需要配置 DDR 初始化
```

### 4. GPIO 和引脚映射

**引脚数量对比:**
| 类型 | RV1106 | RK3562 |
|------|--------|--------|
| GPIO | 32个 | 108个 |
| UART | 3路 | 5路 |
| I2C | 5路 | 6路 |
| SPI | 1路 | 1路 |
| CAN | 无 | 2路 |

**注意**: 引脚定义完全不同，需要重新设计硬件接口！

## RK3562 特定学习

### 需要新学习的知识点

#### 1. ARM 64位架构 (AArch64)

**与 32位 ARM 的差异:**
```asm
; 32-bit ARM (RV1106)
MOV r0, #1          ; 32位寄存器
LDR r1, [r2, #4]    ; 32位地址

; 64-bit AArch64 (RK3562)
MOV x0, #1          ; 64位寄存器 (x0-x30)
LDR x1, [x2, #8]    ; 64位地址
                    ; 支持更大的内存寻址 (>4GB)
```

**C/C++ 代码差异:**
```c
// 32-bit (RV1106)
size_t ptr_size = sizeof(void*);  // 4 bytes
unsigned int address = 0xFFFFFFFF;

// 64-bit (RK3562)
size_t ptr_size = sizeof(void*);  // 8 bytes
unsigned long address = 0xFFFFFFFFFFFFFFFF;
```

**常见问题:**
```c
// 问题: 指针和 int 转换
int* ptr = malloc(100);
int addr = (int)ptr;  // 32位 OK，64位警告/错误

// 解决: 使用 uintptr_t
uintptr_t addr = (uintptr_t)ptr;  // 64位安全
```

#### 2. 多核并行编程

**OpenMP 示例:**
```c
#include <omp.h>

// RV1106: 单核，无法并行
// RK3562: 4核，可以并行

void parallel_process() {
    #pragma omp parallel for num_threads(4)
    for (int i = 0; i < 4; i++) {
        // 4个任务并行执行
        // Core 1: VAD
        // Core 2: ASR
        // Core 3: Chat
        // Core 4: TTS
    }
}

// 编译
gcc -fopenmp -o test test.c
```

**pthread 绑定核心:**
```c
#include <pthread.h>
#include <sched.h>

void* thread_func(void* arg) {
    int core_id = *(int*)arg;
    
    // 绑定到特定核心
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(core_id, &cpuset);
    pthread_setaffinity_np(pthread_self(), sizeof(cpuset), &cpuset);
    
    // 任务执行...
}

// 创建4个线程，分别绑定到4个核心
for (int i = 0; i < 4; i++) {
    pthread_create(&threads[i], NULL, thread_func, &i);
}
```

## 常见问题与解决

### Q1: 交叉编译报错 "undefined reference"

**原因**: 64位和32位库混用  
**解决:**
```bash
# 清理之前的 32位编译产物
make clean
rm -rf build/

# 重新配置为 64位
mkdir build && cd build
cmake .. -DCMAKE_TOOLCHAIN_FILE=../toolchain-rk3562.cmake
```

### Q2: 程序在 RK3562 上运行报错 "cannot execute binary file"

**原因**: 编译成了 32位 ARM 而不是 64位  
**检查:**
```bash
file main
# 错误: main: ELF 32-bit LSB executable, ARM, EABI5 version 1 (SYSV)
# 正确: main: ELF 64-bit LSB executable, ARM aarch64, version 1 (SYSV)
```

### Q3: YOLOv5 模型在 RK3562 上精度比 RV1106 差

**原因**: 可能使用了 INT8 量化，而 RK3562 支持 FP16 精度更高  
**解决:**
```python
# 使用 FP16 量化
rknn.config(
    quantized_dtype='float16',  # 改为 FP16
    ...
)
```

### Q4: 4个 CPU 核心只有一个在工作

**原因**: 默认可能只启用了 1 个核心  
**解决:**
```bash
# 检查核心状态
cat /sys/devices/system/cpu/online
# 应该显示: 0-3

# 如果显示 0，手动启用其他核心
echo 1 > /sys/devices/system/cpu/cpu1/online
echo 1 > /sys/devices/system/cpu/cpu2/online
echo 1 > /sys/devices/system/cpu/cpu3/online
```

### Q5: NPU 推理时报错 "init failed"

**原因**: NPU 驱动未加载或版本不匹配  
**解决:**
```bash
# 检查驱动
lsmod | grep rknpu
# 如果没有输出，加载驱动
insmod /lib/modules/$(uname -r)/kernel/drivers/rknpu/rknpu.ko

# 检查设备节点
ls /dev/rknpu0
# 应该存在

# 更新 RKNN 库版本
# 确保使用与开发板 SDK 匹配的 librknnrt.so
```

### Q6: 网络连接不稳定 (GbE)

**原因**: 可能需要调整网卡参数  
**解决:**
```bash
# 检查网卡状态
ethtool eth0

# 调整接收缓冲区
ethtool -G eth0 rx 4096

# 设置静态 IP (避免 DHCP 问题)
cat /etc/network/interfaces
auto eth0
iface eth0 inet static
    address 192.168.1.100
    netmask 255.255.255.0
    gateway 192.168.1.1
```
